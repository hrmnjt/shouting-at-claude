# Changelog Publishing Workflow

Changelogs for a data platform — new tables, deprecations, schema changes,
bug fixes — need to reach multiple audiences in multiple formats: email to
stakeholders, PDF for records, a website for reference. This document
captures the tooling options for publishing from a single Markdown source.

## The Problem

Writing a changelog in Markdown is straightforward. Publishing it is not:

- **Email (Outlook)** — requires HTML with inline CSS; Outlook uses Word's
  HTML renderer which ignores `<style>` blocks and external stylesheets
- **PDF** — Markdown has no native PDF output; needs an external processor
- **Website** — needs a build pipeline and hosting

The current workaround (Obsidian → obscure copy-to-HTML plugin → paste
into Outlook) handles only the email channel and breaks down as soon as
you need PDF or a website.

## Why Outlook Needs Inline CSS

Standard HTML generated by any tool looks like this:

```html
<style>
  .warning { color: red; font-weight: bold; }
  table { border-collapse: collapse; }
</style>

<p class="warning">Breaking change</p>
```

Outlook ignores the `<style>` block. It only respects styles written
directly on the element:

```html
<p style="color: red; font-weight: bold;">Breaking change</p>
```

A CSS inliner takes the first form and produces the second automatically.

## Immediate Solution: Pandoc + Juice

For today's problem — Markdown file to Outlook-ready email — two tools:

**Install once:**

```bash
brew install pandoc
npm install -g juice
```

**Use per file:**

```bash
pandoc changelog.md --standalone -o changelog.html
juice changelog.html email-ready.html
open email-ready.html   # browser → Select All → Copy → paste into Outlook
```

Wrapped as a script:

```bash
#!/bin/zsh
# md2email.sh
pandoc "$1" --standalone -o /tmp/md2email.html && \
juice /tmp/md2email.html /tmp/md2email-ready.html && \
open /tmp/md2email-ready.html
```

Usage: `./md2email.sh changelog.md`

### What Each Tool Does

**Pandoc** — converts Markdown to HTML (and many other formats). The
`--standalone` flag wraps the output in a full HTML document with a
`<style>` block. Single binary, no dependencies.

**Juice** — takes HTML with `<style>` blocks, computes which CSS rules
apply to which elements, writes them inline, strips the `<style>` block.
The output is Outlook-safe.

## The No-Tooling Option: Ask Claude

For occasional one-off conversions with no tooling setup:

1. Paste the Markdown content to Claude
2. Ask: "convert this to Outlook-ready HTML with inline styles"
3. Save the output as `.html`, open in browser
4. Select All → Copy → paste into Outlook

No install required. Works for infrequent use; a script is better for
regular publishing.

## Future State: Quarto for All Three Outputs

When the need expands to PDF + email + website from a single source,
Quarto is the right tool.

```
changelog.qmd
    ├── quarto render --to html    → email (+ juice for CSS inlining)
    ├── quarto render --to pdf     → PDF via Typst (no LaTeX needed)
    └── quarto publish             → website
```

Quarto files are Markdown with a YAML header — minimal learning curve.
It has native callout blocks well-suited to changelogs:

```markdown
:::{.callout-warning}
## Breaking Change
`user_events` schema changed — `event_type` column renamed to `type`
:::

:::{.callout-note}
## New Table
`session_metrics` added. See data dictionary for schema.
:::
```

Quarto is not the immediate solution (requires setup and buy-in) but is
the natural upgrade path once the Pandoc + Juice pipeline proves its value.

## Decision Guide

| Situation | Approach |
|---|---|
| One-off email today, no setup | Paste to Claude, get HTML back |
| Regular email publishing | `md2email.sh` script (Pandoc + Juice) |
| Email + PDF | Pandoc for both (`--to html`, `--to pdf`) + Juice |
| Email + PDF + website | Quarto |

## Notes on the Email Step

Regardless of the tool used upstream, the email step always requires:

1. Generate HTML
2. Inline the CSS (Juice, Premailer, or equivalent)
3. Open in browser → Select All → Copy → paste into Outlook compose

Alternatively: skip HTML email entirely — attach the PDF, write two
sentences in the email body. Simpler, and the PDF is more readable than
most HTML emails anyway.
